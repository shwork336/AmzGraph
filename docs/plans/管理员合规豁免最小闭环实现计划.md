# 管理员合规豁免最小闭环实现计划

> 本计划承接已完成的合规 FAIL 资产导出拦截最小闭环。
> 本阶段目标是允许管理员对 `FAIL` 图片资产进行强制豁免并记录原因，豁免后的资产可进入 ZIP 导出，同时 manifest 和合规报告保留豁免信息。

## 1. 当前基线

已完成：

- ZIP 导出前会检查最终图片版本下的全部图片资产。
- 任一图片资产 `complianceStatus = FAIL` 且没有豁免能力时，导出返回 `TASK_STATUS_INVALID`。
- `FAIL` 拦截发生在创建导出记录之前。
- `WARNING` 资产允许导出并在 manifest/report 中标记。
- `ImageAsset` 已有：
  - `complianceReviewedBy`
  - `complianceReviewedAt`

当前未完成：

- 没有合规豁免原因字段。
- 没有管理员豁免请求 DTO。
- 没有管理员豁免响应 DTO。
- 没有豁免应用服务。
- 没有豁免 HTTP 接口。
- 导出服务不识别豁免后的 `FAIL` 资产。
- manifest 和合规报告不包含豁免原因。

## 2. 阶段目标

本阶段完成后，应满足：

- 管理员可以对单个图片资产记录合规豁免。
- 只允许对 `FAIL` 图片资产执行豁免。
- 豁免必须记录：
  - 管理员账号
  - 豁免原因
  - 豁免时间
- 豁免后的 `FAIL` 资产允许导出。
- 未豁免的 `FAIL` 资产仍阻止导出。
- manifest 和 `compliance_report.md` 包含豁免人、时间和原因。
- 查询图片资产时返回豁免信息。
- API 不直接暴露 JPA 实体。
- 完整测试通过。

## 3. 范围边界

### 3.1 本阶段做

- 扩展 `ImageAsset`：
  - 新增 `complianceReviewReason`
- 扩展 `ImageAssetResponse`：
  - 返回 `complianceReviewReason`
- 新增 DTO：
  - `ApproveImageAssetComplianceRequest`
  - `ImageAssetComplianceReviewResponse`
- 新增应用服务：
  - `ImageAssetComplianceService`
- 新增接口：
  - `POST /api/v1/listing/{taskId}/versions/image/{imageVersionId}/assets/{assetId}/compliance/approve`
- 修改导出服务：
  - `FAIL` 资产如果已有 `complianceReviewedBy`、`complianceReviewedAt` 和 `complianceReviewReason`，允许导出。
  - manifest/report 写入豁免信息。

### 3.2 本阶段不做

- 不做权限系统。
- 不做撤销豁免。
- 不做批量豁免。
- 不做 `WARNING` 人工确认流。
- 不做真实合规检测。
- 不做管理员审计表拆分。

## 4. 设计原则

- 豁免不改变 `complianceStatus`，保留原始 `FAIL` 结果。
- 豁免是显式管理员动作，必须记录原因。
- 导出服务只识别完整豁免信息，避免半填写字段导致误放行。
- Controller 只负责请求校验和统一响应。
