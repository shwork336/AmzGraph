# 竞品数据基础模块设计

## 1. 目标

在 Listing 任务进入真实 Brief 生成之前，建立标准化、可追溯的竞品数据基础模块。

本阶段先实现手工补录、历史查询、最新快照查询和外部数据源端口抽象。真实 Bright Data 调用、缓存降级和自动 Brief 生成留到后续模块。

## 2. 范围

本阶段实现：

- 新增独立 `competitor` 业务模块。
- 建立标准化 `CompetitorSnapshot` 实体。
- 为外部竞品供应商定义 `CompetitorDataProvider` 端口。
- 手工提交任务已有 ASIN 的标准化竞品快照。
- 相同 ASIN 重复提交时创建新快照版本。
- 查询任务全部竞品快照历史。
- 按 ASIN 查询任务下的最新快照集合。
- 校验任务状态和 ASIN 输入范围。
- 为应用服务和 HTTP 接口补充测试。

本阶段不实现：

- 真实 Bright Data API 调用。
- Bright Data 字段映射配置。
- 原始供应商响应下载和文件保存。
- 快照缓存命中与外部调用重试。
- 自动创建或重新生成 Brief。
- 修改任务最初提交的 ASIN 列表。
- 竞品数据删除或覆盖。
- 前端竞品维护页面。

## 3. 模块边界

新增包：

```text
com.snails.ecommerce.competitor
```

内部按职责拆分：

```text
competitor/domain
competitor/application
competitor/infrastructure
competitor/api
```

`listing` 模块继续负责任务生命周期和 Brief 审核。`competitor` 模块依赖 `ListingTaskRepository` 读取任务状态和原始 ASIN 输入，但不修改任务记录。

## 4. 领域模型

### 4.1 CompetitorSnapshot

字段：

- `snapshotId`：快照业务 ID。
- `taskId`：所属 Listing 任务。
- `asin`：Amazon ASIN。
- `title`：竞品标题。
- `bulletPointsJson`：标准化 Bullet Points。
- `rating`：评分。
- `reviewCount`：评论数量。
- `reviewPainPointsJson`：从评论中整理的用户痛点。
- `keywordSignalsJson`：关键词信号。
- `sourceType`：数据来源类型。
- `sourceName`：具体来源名称。
- `rawResponseFileKey`：原始供应商响应文件键，手工数据为空。
- `capturedAt`：该快照数据的采集时间。
- `createdBy`：手工录入人或系统提供者标识。
- `createdAt`：数据库记录创建时间。

### 4.2 来源类型

新增 `CompetitorSourceType`：

```java
MANUAL
BRIGHT_DATA
CACHE
```

本阶段只创建 `MANUAL` 数据。其他枚举值用于稳定后续数据模型，不表示已经实现对应数据来源。

### 4.3 必填规则

手工补录必须提供：

- `asin`
- `title`
- `createdBy`

以下字段可以为空：

- Bullet Points
- 评分
- 评论数量
- 评论痛点
- 关键词信号
- 来源名称

手工数据保存时：

- `sourceType = MANUAL`
- `sourceName` 为空时统一保存为 `Manual Entry`
- `rawResponseFileKey = null`
- `capturedAt` 由服务端生成

### 4.4 版本规则

- 同一任务、同一 ASIN 可以存在多条快照。
- 每次补录都创建新的 `snapshotId`，不覆盖旧记录。
- 历史顺序按 `capturedAt` 倒序，再按 `snapshotId` 倒序。
- 当前阶段不做内容哈希或重复内容去重。

## 5. 外部数据源端口

定义：

```java
public interface CompetitorDataProvider {
    CompetitorSourceType sourceType();
    List<CompetitorSnapshotData> fetch(String taskId, List<String> asins);
}
```

`CompetitorSnapshotData` 是外部供应商到内部标准化模型之间的应用层数据结构，不是 JPA 实体。

本阶段只定义端口和数据结构，不提供 Spring Bean 实现，也不由手工补录服务调用该端口。这样可以先固定业务层依赖方向，后续 Bright Data 适配器放入 `competitor.infrastructure`。

## 6. 应用服务

新增 `CompetitorSnapshotService`，负责：

- 校验任务存在。
- 校验任务状态为 `WAIT_BRIEF_APPROVE`。
- 解析任务原始 `competitorAsinsJson`。
- 校验补录 ASIN 已存在于任务输入。
- 创建手工快照。
- 查询全部历史。
- 按 ASIN选择最新快照。
- 将实体转换为 API DTO。

服务依赖：

- `ListingTaskRepository`
- `CompetitorSnapshotRepository`
- `IdGenerator`
- Jackson 3 `ObjectMapper`

服务不得：

- 修改 `ListingTask.competitorAsinsJson`。
- 创建 Brief 版本。
- 修改任务或 Brief 状态。
- 调用尚未实现的 Bright Data 适配器。

## 7. API 设计

### 7.1 手工补录

```text
POST /api/v1/listing/{taskId}/competitors/manual
Content-Type: application/json
```

单次请求允许提交一条或多条快照：

```json
{
  "createdBy": "operator@example.com",
  "snapshots": [
    {
      "asin": "B0EXAMPLE1",
      "title": "9 Inch Wireless Car Stereo",
      "bulletPoints": ["Wireless CarPlay", "Android Auto"],
      "rating": 4.4,
      "reviewCount": 325,
      "reviewPainPoints": ["Installation instructions are unclear"],
      "keywordSignals": ["wireless carplay stereo"],
      "sourceName": "Manual Entry"
    }
  ]
}
```

成功时返回本次创建的快照列表。

批量请求采用事务原子性：任意一条数据无效时，整批请求失败，不保存部分数据。

### 7.2 查询全部历史

```text
GET /api/v1/listing/{taskId}/competitors
```

返回任务下全部快照，按 `capturedAt` 和 `snapshotId` 倒序。该接口只读。

### 7.3 查询每个 ASIN 的最新快照

```text
GET /api/v1/listing/{taskId}/competitors/latest
```

每个存在快照的 ASIN 返回一条最新记录。

返回结果顺序按任务原始 `competitorAsins` 顺序排列。尚未补录快照的 ASIN 不返回空占位记录。

## 8. 业务规则

### 8.1 任务状态

手工补录只允许：

```text
ListingTask.status = WAIT_BRIEF_APPROVE
```

任务已经进入 `GENERATING`、`WAIT_FINAL_APPROVE`、`COMPLETED` 或其他状态时，返回 `TASK_STATUS_INVALID`。

查询接口不受任务状态限制，只要任务存在即可读取历史数据。

### 8.2 ASIN 输入范围

- 补录 ASIN 必须已存在于任务提交时保存的 `competitorAsinsJson`。
- ASIN 比较前去除首尾空格并转换为大写。
- 请求中同一 ASIN 不允许重复出现。
- 请求不得自动追加或修改任务原始 ASIN 列表。

ASIN 不在任务输入范围或批次内重复时返回 `INVALID_REQUEST`。

### 8.3 数值字段

- `rating` 为空或位于 `0.0` 至 `5.0`。
- `reviewCount` 为空或大于等于 `0`。

格式和范围错误统一返回 `INVALID_REQUEST`。

## 9. 仓储查询

`CompetitorSnapshotRepository` 至少提供：

```java
List<CompetitorSnapshot> findByTaskIdOrderByCapturedAtDescSnapshotIdDesc(String taskId);
```

最新视图由应用服务从已排序历史中按 ASIN 保留第一条，随后按任务原始 ASIN 顺序输出。

当前阶段不增加数据库窗口函数或原生 SQL。首批竞品数量较小，应用层聚合更容易验证，也避免过早绑定 PostgreSQL 特有查询。

## 10. 错误处理

- 任务不存在：`TASK_NOT_FOUND`
- 任务状态不允许补录：`TASK_STATUS_INVALID`
- ASIN 不属于任务输入：`INVALID_REQUEST`
- 同一请求重复 ASIN：`INVALID_REQUEST`
- 请求字段缺失或数值越界：`INVALID_REQUEST`
- 任务 ASIN JSON 或快照 JSON 无法解析：`INTERNAL_ERROR`

沿用现有 `GlobalExceptionHandler` 和统一 `ApiResponse`，不增加新的错误码。

## 11. 事务与一致性

- 手工批量补录使用单个事务。
- 服务先完成整批校验，再保存所有实体。
- 任意校验或持久化失败时整批回滚。
- 查询接口使用只读事务。
- 快照为追加写入模型，不提供更新操作。

## 12. 测试范围

实体和仓储测试：

- 保存完整手工快照。
- 可空字段允许为空。
- 历史按采集时间和快照 ID 稳定倒序。

应用服务测试：

- 成功批量补录任务已有 ASIN。
- 手工快照来源和审计字段正确。
- 相同 ASIN多次补录创建多条记录。
- 拒绝任务输入范围外的 ASIN。
- 拒绝请求内重复 ASIN。
- 拒绝非 `WAIT_BRIEF_APPROVE` 状态补录。
- 批量中一条无效时不保存任何快照。
- 查询全部历史。
- latest 为每个 ASIN返回最新一条。
- latest 按任务原始 ASIN 顺序输出。
- 查询接口允许读取已进入后续状态的任务。

Controller 测试：

- 三个接口成功响应。
- Bean Validation 错误返回 `INVALID_REQUEST`。
- Service 业务异常保持稳定错误码。

端口测试：

- `CompetitorDataProvider` 和 `CompetitorSnapshotData` 可以独立编译。
- 不要求存在外部供应商 Spring Bean。

## 13. 实施节奏

基础实体、枚举、Repository、DTO 和端口可以成组实现。

涉及业务规则的 Java 文件必须逐个交给用户 review：

1. `CompetitorSnapshotService`
2. `ListingTaskController` 中新增的竞品接口

每个业务文件 review 通过后再继续下一个业务文件。

全部实现完成后运行：

```powershell
cd e-commerce
.\gradlew.bat test
```

完整测试必须输出 `BUILD SUCCESSFUL`。

## 14. 验收标准

- 任务已有 ASIN 可以被手工补录为标准化竞品快照。
- 手工补录不会改变任务原始 ASIN 输入。
- 同一 ASIN 每次补录都会保留新快照版本。
- 任务进入 `GENERATING` 后不能继续补录，但仍能查询历史。
- 可以查询完整历史和每个 ASIN 的最新快照。
- 快照来源、采集时间和创建人可追溯。
- 本阶段不会自动创建 Brief，也不会调用 Bright Data。
- 后续 Bright Data 可以通过统一端口映射到同一快照模型。
- API 不直接暴露 JPA 实体。
- 完整 Gradle 测试通过。
