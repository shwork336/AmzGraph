# 操作审计日志表最小闭环实现计划

> 本计划承接操作人统一审计上下文最小闭环。目标是在关键人工操作发生时写入统一审计日志表，先完成可追踪写入能力，为后续审计查询页和权限系统打基础。

## 1. 当前基线

已完成：

- Controller 支持统一操作人请求头 `X-Operator-Id`。
- 导出取消写入 `canceledBy`、`cancelReason`、`canceledAt`。
- FAIL 合规豁免和 WARNING 确认写入图片资产人工确认字段。
- 服务层已有操作人和原因必填防御。

当前不足：

- 关键人工操作只写在各自业务表上。
- 缺少统一操作审计日志表。
- 后续审计页无法按操作人、动作类型或目标对象统一查询。

## 2. 阶段目标

- 新增 `OperationAuditLog` 实体。
- 新增 `OperationAuditLogRepository`。
- 新增 `OperationAuditLogService`。
- 在以下操作成功后写入审计日志：
  - 导出取消：`EXPORT_PACKAGE_CANCELED`
  - FAIL 合规豁免：`IMAGE_ASSET_COMPLIANCE_APPROVED`
  - WARNING 确认：`IMAGE_ASSET_WARNING_CONFIRMED`
- 审计日志字段至少包含：
  - `auditLogId`
  - `action`
  - `operatorId`
  - `targetType`
  - `targetId`
  - `taskId`
  - `reason`
  - `detailJson`
  - `createdAt`
- 完整测试通过。

## 3. 范围边界

本阶段做：

- 新增实体、仓储、服务。
- 接入导出取消和图片合规人工操作。
- 补服务层测试和实体映射测试。

本阶段不做：

- 不新增审计查询接口。
- 不新增前端审计页。
- 不引入权限系统。
- 不修改已有业务表字段。
- 不保证审计日志失败时业务操作回滚之外的异步容错。
