# 导出超时恢复最小闭环实现计划

> 本计划承接导出任务取消最小闭环。目标是避免导出记录长期卡在 `PENDING` 或 `RUNNING`，让后台 worker 在扫描时先把超时记录标记为失败，给运营一个明确可重试的状态。

## 1. 当前基线

已完成：

- 可创建 `PENDING` 导出记录。
- worker 可扫描并执行 `PENDING` 导出。
- 可取消尚未执行的 `PENDING` 导出。
- 导出列表可按状态筛选。

当前不足：

- `PENDING` 记录如果长期未被 worker 处理，没有自动失败边界。
- `RUNNING` 记录如果进程中断或 worker 异常终止，可能长期卡住。
- 运营无法通过失败重试机制处理卡住的导出记录。

## 2. 阶段目标

- worker 每批执行前先恢复超时记录：
  - 超时 `PENDING` -> `FAILED`
  - 超时 `RUNNING` -> `FAILED`
- 失败原因写入稳定文案，便于前端展示和后续重试。
- 超时阈值由 worker 方法参数传入，默认调度使用 30 分钟。
- 保持 `SUCCEEDED`、`FAILED`、`CANCELED` 不受影响。
- 完整测试通过。

## 3. 范围边界

本阶段做：

- 扩展 `ExportPackageRepository`，支持按状态和创建时间阈值查询超时记录。
- 扩展 `ExportPackageWorkerResult`，增加 recovered 统计。
- 扩展 `ExportPackageWorkerService`，在处理 `PENDING` 前恢复超时记录。
- 补充仓储和 worker 测试。

本阶段不做：

- 不新增 `startedAt`、`updatedAt` 字段。
- 不做分布式锁。
- 不自动重试超时记录。
- 不删除已生成文件。
- 不新增 HTTP 运维接口。
