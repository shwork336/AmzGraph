# 导出失败重试最小闭环实现计划

> 本计划承接已完成的 ZIP 默认交付包最小闭环。
> 本阶段目标是允许运营对失败的 ZIP 导出包发起单独重试，且不影响已 `COMPLETED` 的主任务状态。

## 1. 当前基线

已完成：

- 已完成任务可通过 `POST /api/v1/listing/{taskId}/export` 生成 ZIP 默认交付包。
- 导出包可通过 `GET /api/v1/listing/export/{exportPackageId}` 查询状态。
- `ExportPackage` 已记录：
  - `format`
  - `status`
  - `fileUrl`
  - `manifestUrl`
  - `failureReason`
  - `includedAssetIdsJson`
- ZIP 导出失败时会记录 `FAILED` 和失败原因，不回退主任务状态。
- 同一任务允许多条导出记录。

当前未完成：

- 没有失败导出包重试方法。
- 没有失败导出包重试 HTTP 接口。
- 没有限制只有 `FAILED` 导出包可以重试。
- 没有验证重试时仍基于原任务当前最终选中的图文版本生成新导出记录。

## 2. 阶段目标

本阶段完成后，应满足：

- 只有已存在且 `status = FAILED` 的导出包允许重试。
- 只能重试 ZIP 导出包。
- 重试不覆盖原失败导出包，保留旧记录用于审计。
- 重试成功后创建一条新的 `ExportPackage` 记录。
- 新导出包仍引用任务当前最终选中的 `TextVersion` 和 `ImageVersion`。
- 重试失败不影响主任务 `COMPLETED` 状态。
- 查询接口仍可查询旧失败记录和新成功记录。
- API 不直接暴露 JPA 实体。
- 完整测试通过。

## 3. 范围边界

### 3.1 本阶段做

- 扩展 `ExportPackageService`：
  - 新增 `retryExportPackage(String exportPackageId)`
- 扩展 `ListingTaskController`：
  - 新增 `POST /api/v1/listing/export/{exportPackageId}/retry`
- 补充服务测试：
  - 失败导出包可重试并生成新记录
  - 非失败导出包不可重试
  - 不存在导出包不可重试
  - 非 ZIP 导出包不可重试
- 补充 Controller 测试：
  - 重试成功返回统一 `ApiResponse`
  - 业务错误透传稳定错误码

### 3.2 本阶段不做

- 不做自动重试策略配置。
- 不做重试次数上限。
- 不做异步重试队列。
- 不做 Excel / Markdown / Word 可选导出重试。
- 不做失败原因分类统计。

## 4. 设计原则

- 重试是新的导出尝试，不修改旧失败记录。
- 重试入口只接收旧导出包 ID，不允许前端指定文案版本或图片版本。
- 重试复用 ZIP 默认导出逻辑，避免分叉出第二套打包规则。
- 查询接口保持无副作用。
