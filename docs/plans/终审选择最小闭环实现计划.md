# 终审选择最小闭环实现计划

> 本计划承接已完成的文案生成最小闭环和图片生成最小闭环。
> 本阶段目标是让任务在文案和图片均生成成功后进入终审，并支持运营选择最终 `TextVersion` 和 `ImageVersion`，把任务推进到 `COMPLETED`。

## 1. 当前基线

已完成：

- Brief 人工审核后任务进入 `GENERATING`。
- 文案生成成功后创建 `TextVersion`，并设置 `ListingTask.textStatus = SUCCEEDED`。
- 图片生成成功后创建 `ImageVersion` 和 8 类 `ImageAsset`，并设置 `ListingTask.imageStatus = SUCCEEDED`。
- 文案版本和图片版本均有 `selected` 字段。
- `ListingTask` 已有 `selectedTextVersionId` 和 `selectedImageVersionId` 字段。
- 文案、图片版本查询接口已存在。
- 当前完整测试 `.\gradlew.bat test` 已通过。

当前未完成：

- 文案和图片均生成成功后，主任务仍保持 `GENERATING`，不会自动进入 `WAIT_FINAL_APPROVE`。
- 没有终审选择请求 DTO。
- 没有终审选择响应 DTO。
- 没有终审应用服务。
- 没有最终选择 HTTP 接口。
- 没有防止跨任务选择版本的校验。
- 没有保证同一任务只选中一个文案版本和一个图片版本。
- 没有将任务推进到 `COMPLETED`。

## 2. 阶段目标

本阶段完成后，应满足：

- 文案和图片均成功后，任务进入 `WAIT_FINAL_APPROVE`。
- 运营可以选择一个 `TextVersion` 和一个 `ImageVersion` 作为最终版本。
- 终审选择成功后：
  - `ListingTask.status = COMPLETED`
  - `ListingTask.selectedTextVersionId` 写入最终文案版本 ID
  - `ListingTask.selectedImageVersionId` 写入最终图片版本 ID
  - 被选中的 `TextVersion.selected = true`
  - 被选中的 `ImageVersion.selected = true`
  - 同任务其他文案/图片版本 `selected = false`
- 只能在 `WAIT_FINAL_APPROVE` 状态执行终审选择。
- 被选择的文案版本和图片版本必须属于当前任务。
- 被选择的图片版本必须生成成功。
- API 不直接暴露 JPA 实体。
- 完整测试通过。

## 3. 范围边界

### 3.1 本阶段做

- 扩展 `TextVersionRepository`：
  - 按任务查询全部文案版本。
- 扩展 `ImageVersionRepository`：
  - 按任务查询全部图片版本。
- 新增终审请求 DTO：
  - `ApproveFinalSelectionRequest`
- 新增终审响应 DTO：
  - `FinalSelectionResponse`
- 新增应用服务：
  - `FinalReviewService`
- 新增接口：
  - `POST /api/v1/listing/{taskId}/final/approve`
- 修改文案/图片生成服务：
  - 在当前任务的文案和图片状态都为 `SUCCEEDED` 时推进主任务到 `WAIT_FINAL_APPROVE`。

### 3.2 本阶段不做

- 不做文案或图片人工迭代。
- 不做导出包生成。
- 不做导出文件校验。
- 不做合规豁免审批。
- 不做 `COMPLETED` 后复制新任务。

## 4. 设计原则

- 生成服务只负责把满足条件的任务推进到 `WAIT_FINAL_APPROVE`，不自动选择最终版本。
- 终审选择由独立 `FinalReviewService` 负责，避免继续膨胀 `ListingWorkflowService`。
- 终审选择是事务操作，必须同时更新任务、文案版本和图片版本。
- 选择版本前先完成整体验证，避免部分更新。
- 查询或选择接口都不直接暴露 JPA 实体。
