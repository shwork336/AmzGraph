# 文案生成最小闭环实现计划

> 本计划承接当前已完成的 Listing 提交、Brief 人工审核闭环、竞品手工快照基础模块。
> 本阶段目标是让任务在 Brief 批准后具备可验证的文案生成闭环，先用端口和占位实现跑通状态机、版本保存和查询接口，暂不接入真实 Spring AI Alibaba。

## 1. 当前基线

已完成：

- Listing 任务提交，支持 Markdown 产品资料和 1-4 张产品图上传。
- Markdown 原文读取和 `ProductDocumentExtractor` 端口，占位抽取器已落地。
- `CAR_STEREO` 类目模板初始化与读取。
- `ListingTask`、`ProductRawData`、`ListingBriefVersion`、`TextVersion`、`ImageVersion`、`ImageAsset`、`ExportPackage` 领域实体骨架。
- 任务详情查询。
- Brief 全部版本查询、最新 Brief 查询、人工修改版本、批准 Brief。
- 批准 Brief 后任务进入 `GENERATING`，但文案和图片状态仍为 `NOT_STARTED`。
- 竞品手工快照批量补录、历史查询、latest 查询。
- 当前完整测试 `.\gradlew.bat test` 已通过。

当前未完成：

- `TextVersion` 仓储和 API DTO。
- 文案生成端口与占位实现。
- Brief 批准后触发或显式启动文案生成。
- 文案版本查询接口。
- 生成成功后的 `textStatus` 状态流转。
- 文案生成失败时的状态记录和错误边界。
- 图片生成、终审、导出、真实 AI 接入。

## 2. 阶段目标

本阶段完成后，应满足：

- 系统存在稳定的 `TextVersionRepository`。
- 业务层只依赖 `ListingTextGenerator` 端口，不直接依赖具体 AI SDK。
- 占位文案生成器可以根据已批准 Brief 生成一版可读 `TextVersion`。
- 任务在 `GENERATING` 状态下可生成文案版本。
- 文案生成成功后，`ListingTask.textStatus` 从 `NOT_STARTED` 或 `RUNNING` 推进到 `SUCCEEDED`。
- 可以查询任务下全部文案版本，按创建时间和版本 ID 倒序返回。
- 文案生成不会修改 Brief 历史版本。
- 图片流仍保持 `NOT_STARTED`，主任务暂不进入 `WAIT_FINAL_APPROVE`。
- 完整测试通过。

## 3. 范围边界

### 3.1 本阶段做

- 新增文案生成端口 `ListingTextGenerator`。
- 新增占位实现 `PlaceholderListingTextGenerator`。
- 新增 `TextVersionRepository`。
- 新增文案响应 DTO `TextVersionResponse`。
- 新增应用服务 `TextGenerationService`。
- 新增接口：
  - `POST /api/v1/listing/{taskId}/versions/text/generate`
  - `GET /api/v1/listing/{taskId}/versions/text`
- 补充 service、controller、repository 测试。

### 3.2 本阶段不做

- 不接入真实 Spring AI Alibaba。
- 不做流式生成。
- 不做文案人工迭代和最终选择。
- 不做图片生成。
- 不做 `WAIT_FINAL_APPROVE` 主状态流转。
- 不做导出。
- 不引入 MQ 或复杂异步调度。

## 4. 设计原则

- 业务层通过端口调用文案生成能力，便于后续替换真实 AI 实现。
- `TextVersion` 只保存生成结果和版本关系，不直接暴露 JPA 实体给 API。
- 本阶段只允许基于已批准的最新 Brief 生成首版文案。
- 生成命令保持幂等边界清晰：每次调用创建一个新的 `TextVersion`，不覆盖历史。
- 任务必须处于 `GENERATING`，且 Brief 必须已批准。
- 生成失败只影响 `textStatus`，不回退主任务状态。

- [x] **步骤 2：编写成功路径测试**

验证：

- 调用生成后创建一条 `TextVersion`。
- `TextVersion.taskId`、`briefVersionId` 正确。
- `ListingTask.textStatus` 更新为 `SUCCEEDED`。
- `ListingTask.imageStatus` 保持 `NOT_STARTED`。

- [x] **步骤 3：编写失败和边界测试**

验证：

- 任务不存在返回 `TASK_NOT_FOUND`。
- 任务不在 `GENERATING` 返回 `TASK_STATUS_INVALID`。
- Brief 未批准返回 `TASK_STATUS_INVALID`。
- 最新 Brief 不存在返回 `INVALID_REQUEST`。
- 文案生成器抛异常时，`textStatus` 更新为 `FAILED`，并抛出稳定业务异常。

- [x] **步骤 4：实现 `TextGenerationService`**

职责：

- 查询任务。
- 校验任务主状态和 Brief 状态。
- 获取最新已批准 Brief。
- 获取类目模板。
- 设置 `textStatus = RUNNING`。
- 调用 `ListingTextGenerator`。
- 补齐 `versionId`、`taskId`、`briefVersionId`、`parentVersionId`。
- 保存 `TextVersion`。
- 设置 `textStatus = SUCCEEDED`。
- 查询文案版本列表。

`TextGenerationService.java` 属于业务逻辑文件，完成后先提交给用户 review，确认后再继续 Controller 文件。
